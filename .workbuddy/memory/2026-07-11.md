# 2026-07-11 项目分析：个人网站/第二大脑选型

## 任务背景
用户要长期维护一个"第二大脑"式个人网站：Markdown 驱动（Obsidian 风格，含大量代码/数学/图表），记录心情/生活/学习。
两个 GitHub clone 在 `github-clones/`：`Firefly-hyde`、`Kirameku`；`personal_materials/` 是用户自己的资料（简历/证书/about）。
部署目标（用户已定）：**Cloudflare Pages**。

## 两项目深挖结论
### Firefly-hyde（Astro 7 静态主题，基于 fuwari）
- ✅ 已内置：KaTeX 数学(remark-math+rehype-katex+mhchem)、Expressive Code 代码高亮、Mermaid+PlantUML 图表、Obsidian 风格 callouts(rehype-callouts, 可切 obsidian 主题)、MDX、Pagefind 全文搜索。
- ✅ 内容即仓库 Markdown 文件（`src/content/posts|moments|spec|ziyuan`），纯静态输出 `dist/`，Cloudflare 静态托管零服务器；构建无需 API key。
- ✅ 已有生活类页面开关：gallery(照片墙)、diary(心情/日记)、projects、timeline、moments(朋友圈)、friends 等（`siteConfig.pages.*`）。
- ❌ 缺：Obsidian 灵魂功能 `[[wikilink]]` / `![[embed]]` / 反向链接 / 数字花园集合。需新增 notes 集合 + 1~2 个 remark 插件 + backlinks 组件（可参考已有 getRelatedPosts）。
- 冷门技术（Svelte/Swup/自定义 remark 插件）均为内部/可选，内容作者只写 .md + 改 `src/config/*.ts`，几乎不碰。
- 废弃依赖 `satteri`/`@astrojs/markdown-satteri` 可删。

### Kirameku（Next.js16+React19 前端 / FastAPI+SQLModel+Postgres+OSS 后端 / Vue 后台 / 可选 Kotlin reader）
- ❌ **致命短板**：文章 Markdown 渲染用 `marked`+`highlight.js`，**无 KaTeX、无 Mermaid、无 wikilink、无 callout** —— 与用户"代码+数学+Obsidian"核心需求直接冲突；要支持需把渲染管线重写成 unified+remark-math+rehype-katex+mermaid+wikilink（大工程）。
- DB 实际本地跑 SQLite（18 张表，极朴素，JSON 存 TEXT）→ D1 迁移近无损；OSS→R2 1:1（仅 upload.py 一处）。
- 真正依赖 DB 的只有：文章/评论/留言/点赞/浏览量/仪表盘/访客；说说/相册/项目/友链/收藏早已是 `data/*.ts` 静态文件。
- 维护重：后端+DB+Next 前端+Nginx+OSS(+Kotlin reader)，即使迁 Cloudflare(D1+R2+Worker) 仍需养 Worker+前端构建。
- `Obsidian-Notes/`（Diary/Knowledge-tree/考研）是 **git 子模块，当前未检出真实文件**，导入前需先拿到真实笔记。

## 毛玻璃"看不出效果"的根因与修复（重要教训）
- 用户取色反馈：light #eceef5→#eaeff5、dark #0b0d13→#090e13，几乎无变化。取色位置是**主背景 --page-bg 那块纯色**（非壁纸、非卡片）。
- 根因：
  1. 默认 backgroundWallpaper.mode = "fullscreen"，远程随机图 https://t.alcy.cc/pc 在预览常加载失败 → 退回纯色 --page-bg；且 fullscreen 有个 z-10 壁纸层盖住 <html> 背景，把叠在 html 上的极光也挡了。
  2. 第一版极光渐变 alpha 太低(0.18~0.20)，几乎不可见。
  3. 选择器 bug：`:root.dark html` 永不匹配（:root 就是 html，无 html 后代）→ 暗色极光从没生效。应写 `html.dark`。
  4. 关键认知：**取色器/肉眼都无法从"纯色背景后面的 backdrop-filter blur"感知玻璃**——毛玻璃必须背后有丰富色彩才显形。
- 修复（决定性重做，已构建通过）：
  - backgroundWallpaper.ts: mode "fullscreen"→"none"（弃远程随机图，改自托管极光，Apple 且稳定；可逆）。switchable:true 时 wallpaper-wrapper 仍渲染但 none 下 display:none，不挡背景。
  - glass.css 重写：body::before(静态多色网格,fixed,z-index-2,bg=--page-bg) + body::after(两团大色斑,blur30px,animation aurora-drift 34s 漂移,z-index-1)；light/dark 各一套(用 html.dark)；卡片/浮层/下拉/按钮/navbar 统一 blur+saturate+高光边+分层阴影；prefers-reduced-motion 降级。
  - 验证：pnpm build 通过；dev 已 data-wallpaper-mode="none"，glass.css 含 aurora-drift。
- 关键事实：wallpaper-transparent 仅 overlay 模式加（非 fullscreen）；卡片 .card-base 用 --card-bg，仅 wallpaper-transparent 下强制 --card-bg-transparent(main.css:256 !important)。

## Glass 效果设置面板 Bug 修复（重要）

## Glass 效果"完全无效"的最终根因修复（关键！）
- 用户反馈：设置面板滑块值在变（72%/20.9x/25px），但**卡片完全没反应**，依然是纯白，刷新也没用。无论调多少都无效。
- **根因（真正的根因）**：`glass.css` 是 `.css` 纯 CSS 文件（由 `main.css` 的 `@import './glass.css'` 引入，Vite 当作普通 CSS 处理）。但我在里面用了 **Stylus 专属函数 `unquote("...")`** 包裹所有 `rgb()` 值。浏览器不认识 `unquote()` → 整个 `background-color: unquote(...)` 声明被当作**无效值静默丢弃** → 所有卡片的半透明背景全部失效。
- **修复**：去掉所有 `unquote()` 包装。现代浏览器原生支持 `rgb(255 255 255 / var(--glass-opacity))` 语法，不需要任何包装函数。
- **教训**：`.styl` 文件里用 Stylus 函数没问题；但 `.css` 文件必须只用标准 CSS 语法。以后新增的 glass.css 应改为纯 CSS 写法（或改名为 .styl 并在 Layout.astro 中引入）。
- 验证：dev server 已提供无 unquote 的正确 CSS，构建跑完后确认。

## 14:52 - 光标"开启后系统与自定义光标同时消失"的终极根因（关键 bug！）
### 用户反馈
仅开启自定义光标 → 光标完全没有（系统的也没了）；关闭该选项 → 恢复正常系统光标。
### 根因（真正的元凶）
onMount 里 `const isTouch = "ontouchstart" in window || navigator.maxTouchPoints > 0; if (!isTouch) { 绑定所有鼠标监听 }`。
用户是 **Windows** 设备，很多 Windows 触摸屏/精密触控板机型 `navigator.maxTouchPoints > 0` → `isTouch` 误判为 true → **mousemove 等鼠标监听整段没绑定**。
结果：sync() 照常给 `<html>` 加 `cursor:none`（系统光标被隐藏），但 `mouseX/mouseY` 永远停在初始 -200，自定义光环被永久钉在屏幕外 → 系统 + 自定义光标同时不可见。关闭时移除 cursor:none，系统光标回来。
### 修复（稳健重写 v3）
1. **删除 isTouch 硬性禁用**：鼠标监听**始终绑定**；改为监听 `touchstart`，仅当发生真实触摸时临时 `usingTouch=true` 隐藏自定义光环并交回系统光标，下次 mousemove 自动恢复。
2. **设置同步/事件监听移出所有条件分支**，任何设备都能开关。
3. **开启瞬间定位到视口中心**：`enabled && !hasMoved` 时把 mouseX/mouseY/ringX/ringY 设为视口中心 + `placeInstant()`，不必先移动鼠标就可见。
4. **可见性双保险**：`updateVisibility()` 同时写 `data-active` 属性和直接 `style.opacity`，杜绝 scoped CSS 时序问题。
5. z-index 提到 2147483646/2147483645 顶层。
### 教训
- **绝不能用 `navigator.maxTouchPoints > 0` 判定"是触摸设备"来禁用鼠标逻辑** —— Windows 触摸屏/触控板普遍误报。正确做法：鼠标逻辑常驻，用 `touchstart` 事件切换到触摸模式。
- 自定义光标"看不见"排查顺序：①系统光标是否被 cursor:none 藏了 ②自定义元素坐标是否被更新（监听是否绑定）③opacity/z-index。

## 15:43 - 点击涟漪"彻底没效果"的真根因（Svelte scoped 样式 tree-shake）★★关键
### 用户反馈
确认「点击涟漪」开关已开、硬刷新过，但**仍然完全看不到点击效果**——只有自定义光标和拖尾在，点击涟漪就是没有。明确要求：先找根源，别急着写代码。
### 根因（编译产物取证，非猜测）
Svelte 组件 `<style>` 是 **scoped（作用域化）** 的：编译器只给**模板里出现的元素**自动加 `svelte-xxxx` 哈希属性，并据此限定选择器。而涟漪 div 是用 `document.createElement("div")` 在**运行时**创建的，身上**没有** `svelte-xxxx` 属性。
→ 编译器认为 `.cursor-ripple` 这个选择器在模板中无匹配元素，直接把它（连同 `@keyframes` 引用）当作死代码 **tree-shake 丢弃**。
→ 运行时创建的 div 拿到零样式（0×0、无背景、无动画），等于隐形盒子。所以 `spawnRipple` 其实执行了，但视觉上"什么都没有"。
**取证**：`grep` 编译产物 `dist/_astro/*.css`，发现 `.cursor-ripple-layer.svelte-kqmxog`（容器在模板里→有哈希→正常），而 `.cursor-ripple{...}` 规则**完全不存在**；`@keyframes cursor-ripple-anim` 虽在但无引用者。
### 修复
将 `.cursor-ripple` 规则与 `@keyframes cursor-ripple-anim` 从组件 `<style>` 移到**全局** `src/styles/glass.css`（纯 CSS）。全局 CSS 不 scoped，运行时元素也能匹配。
**验证**：重新 build 后 `grep` 产物，`.cursor-ripple{...}` 已作为**全局规则（无 `.svelte-` 哈希）** 出现，`keyframes cursor-ripple-anim` 全局存在。
### ★通用铁律（务必牢记）
- **任何在运行时 `createElement` 创建、再 `appendChild` 的 DOM 元素，其样式绝不要写在 Svelte/Vue 组件的 scoped `<style>` 里**——会被 tree-shake 丢弃导致不可见。
- 这类动态元素的样式必须放在**全局 CSS**（如 glass.css）或用 `:global(...)` 包裹。
- 排查"动态创建的元素样式不生效"时，直接 `grep` 编译产物 CSS 确认规则是否还在、是否带 scoped 哈希，比反复改 JS 高效得多。
- 构建验证：pnpm build 零错误通过；dev server 200

## 加载慢排查（21:00 起）→ 根因=字体未子集化
### 现象
- 用户反馈：首次打开内容要 10-15 秒才出来（空有 "hyde blog" 标题），换浏览器又慢（无缓存）
- 之前从未这么慢 → 用户质疑"是不是字体原因？但之前没遇到过"

### 根因（关键经验）
- `pnpm build` 是完整流水线：`generate-icons → generate-lqips → astro build → subset-fonts(字体子集化) → pagefind`
- 我前几次构建设了 120-180s 超时，在 `astro build` 阶段就被截断 → **subset-fonts 从未执行**
- 结果：`dist` 一直是 **9.7MB Chikushi-A-maru 全字库**（网站主字体，正文+banner+导航都用），且被 Astro 自动 `preload` → **阻塞首屏渲染**，新浏览器无缓存要下 9.7MB 才出内容
- 用户之前体验快 = 那会儿产物是子集化后的小字体
- **结论：不是配置回退/加了新东西，是我这边构建没跑完整**

### 修复
1. **FontSetup.astro**：local provider 字体不再 preload（fontsource 等小字体仍 preload）——立刻解除首屏带宽抢占
2. **subset-fonts.ts**：原替换逻辑找 `__SUBSET_FONT_xxx__` 占位符，但 Astro 生成的 @font-face 用自身 hash URL，对不上 → 改为按 family 名匹配 Astro 实际生成的 @font-face/src，替换为子集文件并删 9.7MB 原字体
3. 跑**完整** build（后台，约 10+ 分钟，含 pagefind 大站点索引）

### 验证（完整构建后）
- 主字体 Chikushi：**9.7MB → 286KB**（压 97%）✅
- index.html 字体 preload 链接：**1 条 9.7MB → 0 条** ✅
- dist 总体积仍 106M，但大头是：covers 16M(懒加载) + pio 15M(Live2D看板娘,异步) + posts 9.7M + blog 9.7M + assets 15M —— 均非首屏渲染阻塞
- **重要**：用户测的 `localhost:4321` 是 dev 模式（几百 ESM 模块现编译，新浏览器无缓存必慢），不能代表真实速度；应测 `pnpm preview`(生产构建)
- 已启生产预览：**http://localhost:4173/** HTTP 200，让用户用新浏览器/无痕实测

### 经验教训（重要）
- **Astro 项目改完必须跑完整 `pnpm build`**，不能只看 `astro build` 或中途超时截断——subset-fonts/pagefind 等后处理会漏掉
- 设构建超时要给足（完整构建大站点 10+ 分钟），或用 background 模式等完成通知
- 字体 preload 是首屏卡顿头号嫌疑，优先排查 `dist/index.html` 的 `<link rel=preload as=font>`
