## 阅读页三栏独立滚动修复
### 根因
- `.reader-sidebar` 和 `.reader-toc` 都带了 `card-base` class，而 `.card-base { overflow: hidden }`（main.css:209）会**杀死 `position: sticky`**。
- 之前用 `max-height` 而非 `height`，元素内容短时收缩导致 sticky 无足够空间生效。
### 修复（第一版 - 无效）
- 两处阅读页 CSS：`max-height` → `height: calc(100vh - 7rem)`；`overflow: auto` → `overflow-y: auto`
- MainGridLayout 全局样式加 `[data-reading-view="true"] .reader-sidebar/.reader-toc { overflow-y: auto !important }` 确保 card-base 的 hidden 被覆盖。
- **❌ 这版没用** —— sticky 仍飞走。

### 真正根因（对照原始 Firefly-hyde 才看出来）
- 原始 Firefly 的右栏 TOC 是 `#main-grid` 的**直接 grid 子项**（`#right-sidebar-static` > `<SideBar>` > `.sidebar-sticky-section`），`align-items` 默认 `stretch`（栅格项撑满整列全高），`position: sticky` 加在**内部 `.sidebar-sticky-section`** 上 —— 只有在"撑满全高的栅格项"里 sticky 才有空间生效。
- 我之前写 `.reader-layout { align-items: start }` + `position: sticky` 直接加在**栅格项本身**：`align-items: start` 让侧栏栅格项只跟自身内容一样高，sticky 没有可 sticky 的空间 → 直接飞走。

### 正确修复（对齐原始）
- 去掉 `.reader-layout` 的 `align-items: start`（默认 stretch，栅格项撑满全高）。
- sticky 从栅格项移到**内部 wrapper**：HTML 加 `.reader-sidebar-sticky` / `.reader-toc-sticky` 包裹；CSS 里 `.reader-sidebar/.reader-toc`（栅格项）只设 `overflow: visible`，`.reader-sidebar-sticky/.reader-toc-sticky` 设 `position: sticky; top: 6rem; max-height: calc(100vh - 7rem); overflow-y: auto`。
- 删除之前 MainGridLayout 里那条有害的 `overflow-y: auto !important`（它会让外层栅格项变成滚动容器，反而破坏 inner sticky）。
- 移动端 `@media (max-width:768px)` 的 `position: static` 规则同步改到 inner wrapper。

### 🔍 原始 Firefly-hyde 的 TOC sticky 机制（深度分析，关键！）
**原始做法**（posts/[...slug].astro 不放 TOC，走全局右侧栏）：
- `#main-grid` 有 `grid-rows-[auto_1fr_auto]`，TOC 在 `#right-sidebar-static > <SideBar side="right">`，是 `#main-grid` 的**直接 grid 子项**。
- 右栏 `align-items: stretch`（默认）撑满整列高（= 正文高），内部 `#right-sidebar-sticky`（`position: sticky; top: 1rem`）。
- 整页 body 滚动，`#main-grid` 无 overflow 限制 → sticky 相对于视口生效。

**我之前失败的根因**（不是 align-items，不是 card-base overflow）：
- 我把 `reader-layout` 三栏整体塞进 `#swup-container > #content-wrapper` 的 slot 里。
- `#content-wrapper` 带 `class="onload-animation transition-leaving"`，`.onload-animation { animation: fade-in-up ... forwards }`，关键帧 `transform: translateY(2rem) → translateY(0)`。`forwards` 保留 `transform: translateY(0)`。
- **`transform` 创建新的 containing block**，使嵌套在 `#content-wrapper` 内的 sticky 元素定位参考被干扰 → sticky 失效飞走。
- 原始 TOC 在 `#content-wrapper` **之外**（`#right-sidebar` 是 `#main-grid` 直接子项），根本不受此 transform 影响 → 所以原始不飞走。

**正确修复方向（对齐原始，非另辟蹊径）**：
- 把文件树/TOC 从"嵌套在正文 slot 里"提升为 `#main-grid` 的**直接 grid 子项**：文件树 → `#left-sidebar-dynamic`，正文 → `#swup-container`，TOC → `#right-sidebar-dynamic`（这两个 dynamic 容器会被 swup 替换，文章页切换时能正确更新/高亮）。
- 每侧内部用 `position: sticky; top: 1rem` 的 wrapper（照 `#right-sidebar-sticky` 写法）。
- 需改 `MainGridLayout.astro` 的 slot 结构 + 阅读页把内容拆成三块分别填入对应容器。

## 本回合(下午)：部署恢复 + Cloudflare R2 踩坑（关键经验）

### 环境代理坑（致命）
- 本环境**默认设了 HTTP_PROXY/HTTPS_PROXY=127.0.0.1:7890，但该代理没在运行**（连接被拒）。影响：wrangler、curl、Python requests 全都会走代理而失败。
- 解法：wrangler/curl 前 `HTTP_PROXY= HTTPS_PROXY= http_proxy= https_proxy=` 清空；curl 加 `--noproxy '*'`；Python `requests` 设 `s.trust_env=False` 或 pop 掉 env。
- SSH 推送**不走 HTTP 代理**，所以 git SSH 通道不受此影响。

### GitHub 推送在重置型网络下的恢复手法（已验证可行）
- 本环境对大流量上传会"连接重置"（HTTPS push 跑满 2 分钟被 timeout 杀，报错 `unexpected disconnect`/`Broken pipe`）。
- 用户给的 **SSH-over-443** 通道可用且认证通过：`ssh://git@ssh.github.com:443/Muhammad-repos/personal-website.git`（本机已有 `~/.ssh/id_ed25519`）。
- 破坏性提交恢复套路：本机 `git reset --hard <最后一个好提交>`，只 `git add` 极少改动文件（如 5 个函数），提交后 **force-push 一个极小 pack**（远程早有完整 blob，只传新增的少量对象）→ SSH 通道不会被重置。
- 切忌 `git rm -r --cached .` 后直接 commit+push：会把除暂存外所有文件标删并推上 main，制造破坏性提交（本次踩了，已用上述手法恢复）。

### Cloudflare Pages Functions 阴影静态资源（设计性限制）
- 一旦 `functions/notes-assets/[[path]].js` 拦截了 `/notes-assets/*`，Cloudflare 会把这些路径从静态 `ASSETS` 绑定里**摘掉**。`next()` 与 `context.env.ASSETS.fetch(request)` 都**找不到**对应的 Pages 静态副本 → 只能 404，无法回退。
- 结论：**不要用 Pages Functions 去代理/回退静态资源**。若要从 R2 服务静态资源，正确做法是构建期把资源引用 URL 直接改写成 R2 公开地址（`pub-xxx.r2.dev/...`），浏览器直连 R2 CDN，不经过 Functions，也不存在阴影/回退问题。
- 中文路径文件名：curl 必须 URL 编码（`urllib.parse.quote`）才能得到 200，否则假 404（文件其实在）。

### R2 批量上传在本环境不可靠
- `wrangler r2 object put --remote` 单文件可成功（验证过 test.txt 公开可访问）；但 571 个文件批量循环上传很慢且 `wrangler r2 object list` 输出不可信（有时返回 0）。本环境不适合做大体量 R2 批量上传。
- 站点走 Pages 直出已经是在 Cloudflare 全球 CDN 上；对"首页加载慢"的帮助有限——真因是 index.html 828KB + 大 JS/CSS 载荷，应优化载荷而非挪图片 CDN。
